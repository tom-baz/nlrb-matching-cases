# Cluster Assignments Data File

  

This document describes the cluster-assignments file format and how it is generated. The current file in this folder is `cluster_assignments_20260517.csv`; earlier in the project it was named `cluster_assignments.csv`. The structure and creation process described below apply to both.

  

## How the file was created

  

The file was generated by the script `export_cluster_assignments.py` (lives in the upstream clustering project folder). The script combines three data sources:

  

1. **Input CSV** (`blocks_hierarchical_max200_t0.65_merged.csv`) -- the 334,276 company name records produced by the hierarchical blocking stage. Each record has a `block_id` that groups it with other potentially similar names.

  

2. **JSON result files** (`results/blocks_YYYYMMDD_HHMMSS/block_<id>.json`) -- the output of the LLM-CER clustering pipeline. Each file contains a `final_clusters` field: a list of clusters, where each cluster is a list of `company_idx` values that the LLM determined refer to the same entity.

  

3. **SQLite database** (`results/batch_state.db`) -- used only to look up the path to each block's JSON result file.

  

### Script logic

  

1. Load all 334,276 records from the input CSV.

2. Query the SQLite database for the result file path of each completed block (11,328 blocks covering 156,624 records).

3. For each JSON result file, read `final_clusters` and assign each `company_idx` a `cluster_id` (the cluster's index within the block: 0, 1, 2, ...).

4. Left-join the cluster assignments back onto the full input CSV, so every record gets a row.

5. For the 177,652 records whose blocks were not processed by the pipeline (singleton blocks, blocks exceeding the size cap, etc.), assign each record its own sequential `cluster_id` within its block. These records are treated as singletons -- each is its own cluster.

6. Construct a `global_cluster_id` by concatenating `{block_id}_{cluster_id}`, making cluster identifiers unique across the entire dataset.

7. Compute `cluster_size` (how many records share the same `global_cluster_id`) and `is_singleton` (1 if cluster_size is 1, 0 otherwise).

  

### Pipeline overview

  

The clustering pipeline (LLM-CER) that produced the JSON result files works as follows:

  

- **Blocking**: Records are pre-grouped into blocks of similar names using hierarchical blocking with cosine similarity on text embeddings. Blocks larger than 200 records are excluded.

- **NRS (Next Record Set)**: Within each block, records are partitioned into sets of ~12 using k-means on embeddings, then ordered by a similarity chain for LLM friendliness.

- **LLM clustering**: Each set is sent to an LLM (gpt-5-mini) which groups the names into clusters of records that refer to the same real-world entity.

- **CMR (Cluster Merge and Rotation)**: Representatives from each cluster are recombined and sent to the LLM again, allowing cross-set merges. This repeats hierarchically until no more merges are found (two consecutive all-singleton levels).

- **kNN verification**: Before accepting convergence, a kNN-based check compares representatives from different record sets that were never directly compared, closing potential coverage gaps.

  
  

## Column descriptions

  

| Column | Type | Description |

|--------|------|-------------|

| `block_id` | int | Identifier of the block this record belongs to. Records in the same block share a blocking key and were candidates for clustering together. |

| `company_idx` | int | Unique identifier for this company name record across the entire dataset. |

| `company_name` | string | The company name text as it appears in the NLRB data. |

| `block_size` | int | Number of records in this block. |

| `source` | string | Which NLRB dataset this record came from (R Cases or C Cases). |

| `original_block_id` | int | The block ID from the initial (non-hierarchical) blocking stage. |

| `merge_group` | float/null | If this record's block was formed by merging multiple smaller blocks during hierarchical blocking, this holds the merged block's ID. Null if the block was not merged. |

| `original_block_ids` | string/null | Comma-separated list of original block IDs that were merged to form this block. Null if not merged. |

| `cluster_id` | int | Cluster index within the block (0, 1, 2, ...). Only meaningful within a single block. |

| `global_cluster_id` | string | Globally unique cluster identifier: `{block_id}_{cluster_id}`. Use this column to group records that belong to the same cluster. |

| `cluster_size` | int | Number of records in this cluster. A value of 1 means the record is a singleton. |

| `is_singleton` | int | 1 if this record has no cluster partners (cluster_size = 1), 0 if it was grouped with at least one other record. |

  
  

## Grouping name variations by cluster

  

Records that share the same `global_cluster_id` were determined by the LLM to refer to the same real-world entity. For example, the following records all have `global_cluster_id = "1449_0"`:

  

- "dupont"

- "ei dupont de nemours and"

- "e i du pont de nemours and"

- "chemours"

  

These are different name variations -- abbreviations, misspellings, formal legal names -- that the pipeline identified as referring to the same company.

  

To group all name variations together:

  

```python

import pandas as pd

  

df = pd.read_csv("cluster_assignments_20260517.csv")

  

# All multi-record clusters (names that were matched with others)

matched = df[df["is_singleton"] == 0]

  

# Group by cluster and list all name variations

clusters = matched.groupby("global_cluster_id")["company_name"].apply(list)

```

  

Records with `is_singleton = 1` have no matching name variation in the dataset. They either appeared in a block where no other name was similar enough to merge, or they were in a block too small or too large to be processed by the pipeline.
